# Week 0 基盤モジュール説明書

このファイルは、Week 0 でやった「リポジトリ・CMake整備」と「bitset・graph・logger の雛形作成」を、あとから見返せるように整理した記録です。

## 何をしたか

### 1. `CMakeLists.txt` を整えた

- `CMake` で `maximal_clique_bfs` と `maximal_clique_mpi_bfs` を同じ流れで作れるようにした
- OpenMP はある環境だけ有効にする形にした
- 共通ヘッダを `include/` 配下に置き、今後の実装を使い回せるようにした

### 2. `bitset` の雛形を作った

- 可変長のビット集合を扱うための `Bitset` を用意した
- `set`, `reset`, `test`, `count`, `intersects` など、BK 法でよく使う操作をまとめた

### 3. `graph` の雛形を作った

- グラフをファイルから読み込む `Graph` を用意した
- 隣接判定 `has_edge` と、隣接頂点をまとめた `neighbor_bitset` を持たせた
- 今後の `degeneracy ordering` や BK 実装で再利用しやすい形にした

### 4. `logger` の雛形を作った

- CSV を出せる `CsvLogger` を用意した
- 必要なときだけログを出す形にできるようにした
- 再帰木の観測や、後での偏り分析に使えるようにした

## なぜ必要だったのか

### BK 本体をいきなり書かないため

Bron–Kerbosch 法は、再帰・集合演算・枝刈り・探索順の影響が絡むので、最初から本体を全部書くとバグの切り分けが難しくなります。

先に基盤を作ると、あとから

- 正しさの問題
- 性能の問題
- 並列化の問題
- ログの問題

を分けて考えやすくなります。

### 研究で必要なのは「動く」だけではないため

このテーマでは、単にクリークを列挙できるだけでは不十分です。必要なのは、

- 探索木のどこが重いのか
- どの条件で偏りが出るのか
- どう並列に分けるとよいのか

を後から説明できることです。

そのためには、最初からログや共通データ構造を整えておく必要があります。

## それぞれの意味

### `bitset`

BK 法では、候補集合 `P` や除外集合 `X` と、ある頂点の近傍の交差を何度も計算します。`bitset` は、その計算を速くし、実装も統一するための土台です。

### `graph`

入力グラフの読み込み方法や隣接判定がバラバラだと、逐次版・並列版・評価用コードで結果がずれやすくなります。`graph` を共通化することで、研究の比較条件をそろえやすくなります。

### `logger`

この研究では、「速かったか」だけでなく「なぜ遅くなるのか」を見たいです。`logger` は、そのための観測口です。後で再帰木の深さ、候補数、部分木サイズなどを記録する前提になります。

- 研究用の観測基盤として、`logger` を差し込める形にした
