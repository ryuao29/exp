# OpenMP task 化の説明

この文書は、Bron–Kerbosch の浅い深さ task 化をどう入れたか、何を保証していて、どこまでを並列化対象にしているかをまとめたものです。

## 先に読むとよいもの

- [docs/bk_parallel_fundamentals.md](bk_parallel_fundamentals.md)
- [docs/recursion_tree_logging.md](recursion_tree_logging.md)

OpenMP の話は、BK の探索木が見えていないと理解しにくいです。

## 目的

最初の OpenMP 版は、探索木の浅い部分を task に切って、単一ノード上で負荷を分散できるかを確認するためのものです。

この段階の目標は「速い最終版」ではなく、「後で MPI にもつながる分割単位を見つけること」です。

### 修論上の意味

この段階は、単に OpenMP を使う練習ではありません。

- どの深さで分割すべきか
- どの入力で偏りが強いか
- どの粒度で task を切ると得か

を実験的に調べるための基盤です。

## 実装の考え方

- 既存の逐次 BK baseline を壊さず、別の並列エントリポイントを追加した
- task は深さ制限つきで生成する
- 各 task はその時点の `P` と `X` のコピーを持つので、兄弟探索と干渉しない
- クリク報告は callback を `critical` で保護している

対象コード: `include/exp/bron_kerbosch.hpp`

CLI: `src/maximal_clique_bk_omp.cpp`

### なぜ `task` を使うのか

OpenMP には `parallel for` もありますが、BK の探索木は単純なループではありません。再帰的に分岐するので、`task` の方が構造に合っています。

ただし、`task` は万能ではなく、生成しすぎるとオーバーヘッドが増えます。そのため、深さ制限を入れています。

## 重要な制約

- この初版では再帰木 CSV ログは並列版に付けていない
- task の粒度が小さすぎるとオーバーヘッドが勝つので、`task_depth` を 1 から順に試す
- まずは正しさを優先し、性能最適化は後で行う

### ここでの研究課題

この初版で大事なのは、速さそのものよりも「task 化が偏りをどの程度減らすか」を測れることです。

したがって、次のような比較が必要になります。

- 逐次 BK vs OpenMP BK
- task_depth を変えたときの違い
- 入力グラフを変えたときの違い
- ログで見える偏りと実行時間の対応

## 使い方

```bash
./build/maximal_clique_bk_omp graph.txt 2 4
```

- 2 番目の引数: task を生成する最大深さ
- 3 番目の引数: OpenMP スレッド数

### 実験で見るべき値

- 実行時間
- スレッド数ごとの speedup
- task_depth ごとの性能変化
- ログから見た深さ別の仕事量
- 葉ノードまでの分岐数

## これからのつながり

この task 化の結果は、ノートブック `notebooks/recursion_analysis.ipynb` で見た偏りと合わせて評価する。

そのあとで、MPI に流す浅い部分木の切り方を決める。

## 修論に使うなら

OpenMP の成果を修論に乗せるには、単に「並列化できた」ではなく、

- どの特徴量が偏り予測に効いたか
- どの threshold が妥当だったか
- どの入力で改善が再現したか

まで示す必要があります。

ここまでいけば、OpenMP は「実装の練習」ではなく「研究の検証対象」になります。

## 更新ルール

- task の深さ条件や callback の扱いを変えたら、この文書を更新する
- CLI 引数を増やしたら `README.md` も更新する
- 正しさを変える修正を入れたら、対応するテストを増やす
