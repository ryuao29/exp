# 逐次 Bron–Kerbosch baseline 説明書

このファイルは、簡易逐次 BK 実装で何をしたか、なぜそれを先に作ったのか、コードがどう動いているのかを、あとから見返せるようにまとめたものです。

## 先に読むとよいもの

- [docs/bk_parallel_fundamentals.md](bk_parallel_fundamentals.md)
- [docs/degeneracy_ordering.md](degeneracy_ordering.md)

BK の記号や探索木の見方が曖昧だと、この baseline の意味が見えにくくなります。

## 何のために作ったか

この実装の目的は、まず「極大クリーク列挙が正しく動く最小の逐次版」を持つことです。

研究としては、いきなり並列化に行くのではなく、

- 逐次で正しい結果が出る
- 小さい入力でテストできる
- 今後の pivot 追加や degeneracy ordering 追加の土台になる

という状態を先に作る必要があります。

### 修論上の位置づけ

この baseline は、単に「最初のコード」ではありません。後で並列化の良し悪しを判断するときの比較対象です。

そのため、ここでの正しさと実装の単純さが重要になります。

特にこのテーマでは、並列化の前に baseline を固定しておかないと、性能差が「アルゴリズムの差」なのか「実装の差」なのか分からなくなります。

## 今回やったこと

### 1. pivot なしの Bron–Kerbosch を実装した

`include/exp/bron_kerbosch.hpp` に、pivot を使わない再帰版の BK を実装しました。

この版では、各再帰呼び出しで

- 現在のクリーク `R`
- 候補集合 `P`
- 除外集合 `X`

を使って探索します。

`P` が空で `X` も空なら、今の `R` は極大クリークなので、それを出力します。

### この実装で見ておくべき点

- `P` の候補をどう順に試しているか
- 再帰のたびに `P` と `X` がどう変わるか
- 葉ノードの条件が何か
- なぜこの条件で極大クリークだけが出るのか

### 2. CLI 実行ファイルを作った

`src/maximal_clique_bk.cpp` は、グラフファイルを読み込んで極大クリークを 1 行 1 個で出力する実行ファイルです。

これは次の用途を持ちます。

- 手元で動作確認する
- 小さい入力で列挙結果を見る
- 今後の最適化と比較する baseline にする

### 3. テストを作った

`tests/bron_kerbosch_test.cpp` では、小さなグラフに対して期待される極大クリークが出ることを確認しています。

これで、あとから pivot を入れたり、degeneracy ordering を入れたりしても、正しさを壊していないかをすぐ確認できます。

## コードの役割

### `include/exp/bron_kerbosch.hpp`

ここが BK の本体です。

主な関数は次の 2 つです。

- `bron_kerbosch_no_pivot_recursive`
  - 再帰の 1 ステップを担当する
  - `P` の各頂点を 1 つずつ試して、次の `P` と `X` を作る
- `enumerate_maximal_cliques_no_pivot`
  - 初期状態を作って再帰を呼ぶ
  - グラフ全体の頂点を最初の候補集合 `P` に入れる

このファイルの意図は、アルゴリズムの中身を CLI から切り離して、あとで再利用しやすくすることです。

### 逐次 BK を読むときのポイント

この関数を読むときは、コードを上からなぞるだけでなく、次の順に追うと分かりやすいです。

1. 初期状態で `P` に全頂点を入れているか
2. 再帰呼び出しの前に `next_p` と `next_x` を交差で絞っているか
3. 再帰後に `p.reset(vertex)` と `x.set(vertex)` をしているか
4. `P` と `X` が空になったときだけ出力するか

この 4 点が、BK の正しさの核です。

### `src/maximal_clique_bk.cpp`

これは「使う側」です。

やっていることは単純で、

1. グラフを読む
2. `enumerate_maximal_cliques_no_pivot` を呼ぶ
3. 見つかったクリークを表示する

という流れです。

研究では、こういう薄い CLI を作っておくと、後で入出力だけ変えて同じアルゴリズムを使い回せます。

### `tests/bron_kerbosch_test.cpp`

これは正しさの固定点です。

今のテストは、

- 三角形 `0,1,2`
- 辺 `2,3`
- 辺 `3,4`

を含む小さいグラフに対して、極大クリークが `0 1 2`, `2 3`, `3 4` の 3 個になることを確認しています。

このテストがあると、実装を少し変えたときに「列挙漏れ」「重複」「終了条件の破壊」をすぐ見つけられます。

### 研究利用の観点

このテストは最小限なので、研究を進めるなら次のような追加テストも欲しくなります。

- 完全グラフで極大クリークが 1 個になること
- 空グラフで各頂点が単独クリークになること
- 頂点順序を変えても列挙結果が同じ集合になること
- 小さなランダムグラフで逐次版と並列版の結果が一致すること

## なぜこの形にしたか

### まず pivot なしにした理由

pivot を入れると普通は高速になりますが、最初の段階ではコードの見通しが少し悪くなります。

今は研究の最初の土台なので、

- 再帰の流れが読めること
- 候補集合 `P` と除外集合 `X` の動きが追えること
- 正しさをテストしやすいこと

を優先しました。

### ここで避けたかった失敗

最初から pivot や degeneracy を詰め込みすぎると、

- バグの原因が分かりにくくなる
- 逐次と並列の差が見えにくくなる
- ログ設計が後手に回る

という問題が出やすいです。

研究としては、まず「何が遅いのか」を見える化する土台が必要でした。

### `Bitset` を使う理由

BK では集合の交差をたくさん計算します。

今後 `R/P/X` を大きいグラフでも扱いたいので、`std::vector` の走査だけでなく、`Bitset` の形で統一しておくのが有利です。

この baseline ではまだ `Bitset` の全部を使い切っていませんが、次の pivot 版や degeneracy 版にそのままつながります。

## 今回の到達点

- 極大クリークを実際に列挙できる逐次 baseline を持った
- 小さなグラフで正しさをテストできるようにした
- 今後の最適化や並列化と比較する土台を作った

## これからのつながり

この baseline は、次の作業の起点になります。

- `degeneracy ordering` の追加
- pivot の導入
- 再帰木ログの出力

研究としては、ここから「どう速くするか」だけでなく「なぜ偏るか」を見られるようにしていく段階です。

## 修論に使うならどこまで深めるか

この baseline は、修論の本命そのものというより、後続の並列化・評価の土台です。したがって、修論では次を使う前提になります。

- ベースラインの正しさを固定する
- 並列版と同じ入力で比較する
- ログと結果の整合性を確認する
- どの変更が性能や偏りに効いたかを説明する
